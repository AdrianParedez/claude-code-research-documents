# Repo Standards

Last reviewed: 2026-05-21

This repo is easiest to work with when the tree explains itself before you open
anything. These rules exist to make that true.

## Standards Baseline

As of 2026-05-21, the repo uses the latest official beta only where an official
beta actually exists.

- Conventional Commits: `1.0.0-beta.4`
- Semantic Versioning: `2.0.0`
- Google filename guidance: current published living documentation
- GitHub contributor, PR template, and `CODEOWNERS` guidance: current published
  living documentation
- License: Apache-2.0
- Default branch: `main`

That mix is intentional. Conventional Commits has an official beta page.
Semantic Versioning does not. Google and GitHub publish living docs rather than
versioned specs. So the repo follows the newest official source in each case
instead of pretending every standard has a beta track.

## What Comes From Upstream and What Does Not

Upstream standards define:

- commit grammar
- breaking-change semantics
- semantic version meanings
- filename hygiene
- valid locations for `CONTRIBUTING.md`, `pull_request_template.md`, and
  `CODEOWNERS`

Repo policy defines:

- the exact filename grammar used here
- the allowed folder layout
- the allowed commit scopes
- the branch naming pattern
- the public writing standard for repo-facing docs

## Filenames

Public documents should be legible before they are opened. This repo uses:

`<doc-type>--<domain>--<topic>.md`

Rules:

- use lowercase only
- use ASCII only
- use hyphens inside tags
- use double hyphens `--` as tag separators
- use exactly three tags for normal documents
- do not use spaces, underscores, camelCase, dates, `final`, `new`, `v2`, or
  similar filler

Approved `doc-type` tags:

- `paper`
- `brief`
- `audit`
- `notes`
- `spec`
- `template`

Examples:

- `paper--claude--prompting.md`
- `paper--claude-code--prompting.md`
- `paper--claude-code--systems.md`
- `brief--anthropic--writing-style.md`
- `spec--repo--release-process.md`

Reserved exceptions:

- `README.md`
- `CONTRIBUTING.md`
- `CODEOWNERS`
- `pull_request_template.md`
- dotfiles such as `.markdownlint.json`
- GitHub special files inside `.github/`

## Folders

The top level should stay small. If someone has to guess where something goes,
the structure is already too loose.

Folder rules:

- use lowercase ASCII only
- use hyphens, not underscores
- no spaces
- no camelCase
- use plural collection names for durable document groups

Allowed top-level folders:

- `.github/`
- `.husky/`
- `docs/`
- `prompts/`
- `scripts/`

Allowed root files:

- `README.md`
- `CONTRIBUTING.md`
- `LICENSE`
- `.gitignore`
- `.gitattributes`
- `.markdownlint.json`
- `.gitmessage.txt`
- `package.json`
- `package-lock.json`

Current folder layout under `docs/`:

- `papers/` for canonical research papers
- `site/` for the project website and its static preview assets
- `standards/` for repository rules and process documents

Current folder layout under `prompts/`:

- `templates/` for reusable prompt templates grounded in the research corpus

Additional folders such as `briefs/`, `audits/`, `notes/`, or `templates/`
should only be added when they represent a stable document class, not a one-off
convenience.

Use `scripts/` only for repo guardrails and enforcement logic. Do not turn it
into a catch-all utilities folder.

## Public Writing Standard

These docs are public. They should read like technical documents written by a
person with a point of view, not like autogenerated policy output.

Required style:

- start with the real point quickly
- prefer concrete language over abstract management phrasing
- cut repeated framing
- avoid inflated claims and empty rigor words
- keep lists for genuinely list-shaped material
- if a sentence can be removed without loss, remove it

This is the practical test: if a paragraph sounds like it could have been
dropped into any repo without changing a word, rewrite it.

## Commits

The commit history should read like an index, not a diary. This repo uses
Conventional Commits `1.0.0-beta.4` with one stricter local rule: scope is
required for every commit except `revert`.

Required header:

`<type>(<scope>): <description>`

Breaking header:

`<type>(<scope>)!: <description>`

Under `1.0.0-beta.4`, `!` is only an extra signal. It does not replace the
breaking-change footer. If a change is breaking, use both the `!` marker and a
footer:

`BREAKING CHANGE: <details>`

Allowed commit types in this repo:

- `feat`
- `fix`
- `docs`
- `improvement`
- `refactor`
- `chore`
- `test`
- `ci`
- `build`
- `revert`

Allowed scopes:

- `repo`
- `readme`
- `standards`
- `github`
- `lint`
- `papers`
- `claude`
- `claude-code`

Description rules:

- use the imperative mood
- start lowercase unless a proper noun requires otherwise
- do not end with a period
- keep it under 72 characters
- say what changed, not how busy the change felt

Body rules:

- leave one blank line after the header
- use the body when the reason or constraint is not obvious
- explain why, tradeoffs, or follow-on effects when they matter

Allowed footers:

- `BREAKING CHANGE:`
- `Refs:`
- `Co-authored-by:`

Good examples:

- `docs(claude): tighten prompting paper abstract`
- `docs(claude-code): rename systems paper to tagged format`
- `chore(repo): move standards into docs`
- `fix(readme): correct papers directory links`
- `feat(standards)!: require scoped commits for all changes`
- `improvement(repo): simplify contributor checklist language`

Bad examples:

- `updated files`
- `Fix stuff`
- `docs: changes`
- `feat(repo): added a lot of updates to various files in the repository.`

## Enforcement

Policy is not enough on its own. This repo enforces the rules locally and in
CI.

Primary commands:

- `npm ci`
- `npm run lint:md`
- `npm run lint:filenames`
- `npm run lint:commits`
- `npm run check`

Local enforcement:

- `.husky/pre-commit` runs Markdown and repo-policy checks
- `.husky/commit-msg` runs commitlint with the repo's beta-specific rules

CI enforcement:

- `.github/workflows/repo-guard.yml` runs on push and pull request to `main`
- CI is authoritative if local state and CI ever disagree

## Versioning

If you ever tag releases for this repo, use official SemVer `2.0.0`.

- `fix` maps to patch
- `feat` maps to minor
- `BREAKING CHANGE` maps to major

SemVer `2.0.0` also allows pre-release identifiers such as `-beta.1`, but that
does not mean the SemVer specification itself has a newer official beta release
than `2.0.0`. As of 2026-05-21, it does not.

## Branches

Branch names are local policy, not an upstream standard. Use:

`<type>/<scope>-<short-slug>`

Allowed branch types:

- `docs`
- `repo`
- `fix`
- `feat`
- `chore`
- `refactor`

Examples:

- `docs/claude-prompting-cleanup`
- `repo/standards-hardening`
- `fix/readme-links`

## Pull Requests

Use `.github/pull_request_template.md`.

Every pull request should answer five questions clearly:

- what changed
- what part of the repo it touches
- how it was checked
- what issue or context it belongs to
- whether naming and placement rules were checked

Do not bundle unrelated structure work and content edits just because they were
done in the same session. If the changes would be easier to review as separate
pull requests, split them.
